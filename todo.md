# TODO: Rust Components for Three-Body Oracle + Predictor

**Assumptions (explicit so tests are honest)**
- Units are SI by default. Constants `G`, `k_e`, `mu_0` are configurable in code and config so tests can use normalized values.
- Electromagnetism is quasi-static and nonrelativistic. No retardation, no radiation reaction, no self-field effects. This means momentum and energy conservation are not guaranteed once magnetic terms are enabled, and tests must not assume them.
- Softening `epsilon` is optional and changes physics. Tests must separate `epsilon = 0` (physical) from `epsilon > 0` (numerical stability).

**Academic concerns → Mitigations & options (actionable)**
- EM fidelity: keep current quasi-static model as `em_quasistatic`, document its validity limits, and add an explicit `em_off` baseline for “pure gravity” truth testing.
- Conservation interpretation: always label energy as “mechanical proxy,” and gate invariant tests by regime so EM+magnetic never asserts strict conservation.
- Close encounters: formalize a policy with three options (soften, switch integrator, stop) and make it user-configurable and testable.
- Integrator structure under EM: add an EM-suitable integrator option (Boris or implicit midpoint) and require warnings when leapfrog is used with EM.
- Chaotic predictability: add short-horizon and divergence-time diagnostics so we measure what can actually be predicted.
- Discovery stability: if discovery is implemented in Rust, enforce feature-library constraints and add regression tests that detect collinearity/overfit behavior.

7. Implement barycentric initialization in `threebody-core/src/frames.rs`: function `to_barycentric(system) -> system` that subtracts COM position and velocity from each body. Physics requirement: for gravity-only or electrostatic-only (no magnetic terms), total linear momentum should be zero after barycentric conversion and remain zero under internal forces. Tests: COM position and momentum are zero after conversion; gravity-only integration keeps COM near zero.
8. Implement gravity force model in `threebody-core/src/forces/gravity.rs`: acceleration `a_i = sum_{j!=i} G * m_j * (r_j - r_i) / |r_j - r_i|^3`; optional softening replaces `|r|^3` with `(r^2 + epsilon^2)^(3/2)`. Tests: two-body symmetry with `epsilon = 0` gives equal-and-opposite accelerations scaled by masses; gravitational acceleration is zero if all masses are zero.
9. Implement electrostatic and magnetic force model in `threebody-core/src/forces/em.rs`: `E_i = sum_{j!=i} k_e * q_j * (r_i - r_j) / |r_i - r_j|^3`; `B_i = sum_{j!=i} (mu_0 / 4pi) * q_j * (v_j x (r_i - r_j)) / |r_i - r_j|^3`; `a_i_em = (q_i / m_i) * (E_i + v_i x B_i)`; softening same as gravity. Tests: with all charges = 0, EM acceleration is zero; with velocities = 0, magnetic terms are zero and Coulomb forces are equal-and-opposite; with `m_i` nonzero and `q_i = 0`, EM acceleration is zero.
10. Implement potentials in `threebody-core/src/forces/potentials.rs`: `phi_i = sum_{j!=i} k_e * q_j / |r_i - r_j|`; `A_i = sum_{j!=i} (mu_0 / 4pi) * q_j * v_j / |r_i - r_j|`. Tests: with all charges = 0, both potentials are zero; with velocities = 0, vector potential is zero.
11. Create a unified force aggregator in `threebody-core/src/forces/mod.rs`: compute pairwise `r_ij`, `d_ij`, `inv_d3` once per step and pass to gravity/EM to avoid duplication; provide `compute_accel(system, config) -> [Vec3; 3]` and `compute_fields(system, config) -> {E,B,phi,A}`. Tests: aggregation with gravity-only matches gravity module; aggregation with EM-only matches EM module; turning flags on/off yields deterministic outputs.
12. Add `threebody-core/src/regime.rs` to compute regime diagnostics (min pair distance, max speed, max acceleration, and dt ratio). Acceptance criteria: diagnostics are produced for every step, are finite, and are persisted for later reporting. Tests: deterministic diagnostics for a static configuration; no NaNs or infinities.
13. Implement diagnostics in `threebody-core/src/diagnostics.rs`: linear momentum `P = sum m_i * v_i`, angular momentum `L = sum r_i x (m_i * v_i)`, and energy proxy = kinetic + gravitational potential + electric potential. Tests: for gravity-only two-body circular orbit (normalized), energy proxy stays bounded under small dt; EM energy proxy computed equals manual sum for a static configuration.
14. Define config schema in `threebody-core/src/config.rs` using `serde`: constants; integrator choice and tolerances; softening; enable_gravity; enable_em; output toggles. Tests: JSON roundtrip, defaults applied correctly, invalid configs rejected with clear errors.

17. Implement leapfrog (kick-drift-kick) in `threebody-core/src/integrators/leapfrog.rs`. **Cross-cutting test rules (apply to every step)**
1. Every physics formula must appear in code comments or module docs in the same file as its implementation, and every formula must have at least one unit test that checks a numeric example.
2. For invariants, tests must specify the regime: gravity-only and electrostatic-only can assert momentum conservation; EM with magnetic terms must not assert strict conservation.
3. Tolerances must be explicit in tests, stored in `threebody-core/tests/tolerances.rs` to keep DRY.
4. Every public function must have at least one test that covers a failure mode (invalid inputs, division by zero, or `epsilon` edge cases).
5. Every CLI command and runner task must have at least one integration or smoke test that exercises its happy path.
